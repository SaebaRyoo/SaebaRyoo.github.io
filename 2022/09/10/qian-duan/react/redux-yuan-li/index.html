redux源码学习
发表于：2022-09-10 | 分类： 前端

Redux 是一个用于 JavaScript 应用的状态管理库，它的核心原理可以概括为：通过单一数据源、状态只读、纯函数修改这三大原则，结合发布-订阅模式，提供可预测、可追踪的状态管理。

下面我从设计思想、核心概念、工作流程、与 React 的结合以及中间件原理几个层面来展开。

一、三大设计原则

1. 单一数据源

整个应用的状态被保存在一个对象树中，并且这个对象树只存在于唯一一个 Store 中。这使得状态管理变得集中、易于调试和持久化。

2. 状态是只读的

唯一改变状态的方式是触发一个 Action（一个描述“发生了什么”的普通对象）。这种设计保证了视图或任何异步操作都不会直接修改状态，所有的状态变更都集中、有序，便于追踪。

3. 使用纯函数执行修改

描述“状态如何变化”的逻辑写在 Reducer 中。Reducer 是纯函数：(previousState, action) => newState。它接收当前状态和 action，返回全新的状态，而不会修改原状态。纯函数的特性使得时间旅行、热重载等功能成为可能。

二、核心概念与工作流程

Redux 的核心流程是一个严格的单向数据流：

text

View/用户交互 → dispatch(action) → Reducer → newState → View 更新

具体涉及以下角色：

1. Store

Store 是连接所有部分的中心，它：

  • 维护应用的状态树
  • 提供 getState() 获取当前状态
  • 提供 dispatch(action) 触发状态更新
  • 提供 subscribe(listener) 注册监听器，当状态变化时通知

创建 Store 通常通过 createStore(reducer, preloadedState, enhancer)。

2. Action

Action 是一个普通 JavaScript 对象，必须包含 type 字段，用于描述要进行的操作。例如：

javascript

{ type: 'INCREMENT', payload: { amount: 1 } }

Action 也可以由 Action Creator 函数生成，便于复用。

3. Reducer

Reducer 指定状态如何响应 action。由于 reducer 是纯函数，它必须：

  • 不直接修改原状态
  • 不执行副作用（如 API 请求）
  • 对于未知的 action，返回原状态

常见的 reducer 结构会使用 switch 语句根据 action.type 返回新的状态对象。随着应用变大，可以使用 combineReducers 将多个 reducer 合并，每个 reducer 管理状态树的一部分。

4. 数据流示例

javascript

// 1. 定义 reducer
function counter(state = 0, action) {
  switch (action.type) {
    case 'INCREMENT':
      return state + 1;
    case 'DECREMENT':
      return state - 1;
    default:
      return state;
  }
}

// 2. 创建 store
import { createStore } from 'redux';
const store = createStore(counter);

// 3. 订阅更新（通常由视图库如 React-Redux 自动处理）
store.subscribe(() => console.log(store.getState()));

// 4. 分发 action
store.dispatch({ type: 'INCREMENT' }); // 输出 1

三、Redux 的实现原理（简析）

Redux 源码非常精炼，核心实现围绕 createStore 函数。

Store 的内部结构（简化版）

javascript

function createStore(reducer, preloadedState) {
  let currentState = preloadedState;
  let currentReducer = reducer;
  let listeners = [];

  function getState() {
    return currentState;
  }

  function dispatch(action) {
    // 调用 reducer 计算新状态
    currentState = currentReducer(currentState, action);
    // 通知所有订阅者
    listeners.forEach(listener => listener());
    return action;
  }

  function subscribe(listener) {
    listeners.push(listener);
    // 返回取消订阅的函数
    return function unsubscribe() {
      listeners = listeners.filter(l => l !== listener);
    };
  }

  // 初始化时，派发一个 init action 来生成初始状态
  dispatch({ type: '@@redux/INIT' });

  return { dispatch, subscribe, getState };
}

关键点：

  • 状态保存在闭包变量 currentState 中，外部无法直接修改
  • dispatch 是唯一触发状态更新的方式，它调用 reducer 并遍历执行所有监听器
  • 监听器机制是简单的发布-订阅，没有对变化做精细比较，由上层（如 React-Redux）负责优化

四、与 React 的结合

Redux 本身与视图库无关，但常与 React 搭配使用。官方推荐的绑定库是 React-Redux，它通过 Provider 组件和 connect 高阶组件（或 hooks）将 Redux store 注入到 React 组件树中。

React-Redux 的工作方式

  • Provider：利用 React Context 将 store 向下传递给所有子组件。
  • connect：将组件需要的 state 和 dispatch 映射为 props，并监听 store 变化。当 store 更新时，通过 mapStateToProps 筛选组件关心的数据，仅当数据变化时才重新渲染组件。

Hooks 版本则使用 useSelector 和 useDispatch，更加简洁。

五、中间件（Middleware）原理

Redux 本身只支持同步数据流，但通过 中间件 可以处理异步操作、日志等副作用。中间件位于 dispatch 发出 action 和 action 到达 reducer 之间。

中间件的函数签名

javascript

const middleware = store => next => action => {
  // 在 action 到达 reducer 之前做一些事
  // 可以调用 next(action) 传递 action，也可以调用 store.dispatch 再次派发
}

applyMiddleware 的实现原理

applyMiddleware 是一个 Store Enhancer，它通过函数式组合（compose）将多个中间件串联起来，增强 dispatch 方法。核心代码如下（简化）：

javascript

function applyMiddleware(...middlewares) {
  return (createStore) => (reducer, preloadedState) => {
    const store = createStore(reducer, preloadedState);
    let dispatch = () => { throw new Error('...') };

    const middlewareAPI = {
      getState: store.getState,
      dispatch: (action) => dispatch(action) // 最终会被中间件包装的 dispatch
    };
    const chain = middlewares.map(middleware => middleware(middlewareAPI));
    dispatch = compose(...chain)(store.dispatch);

    return { ...store, dispatch };
  };
}

中间件通过嵌套函数层层包裹原始的 store.dispatch，形成一个洋葱模型，可以在 action 流经时做各种处理（如异步请求、日志记录）。

异步中间件示例（redux-thunk）

redux-thunk 是最常用的异步中间件，它允许 action creator 返回一个函数（而不是普通对象），这个函数接收 dispatch 和 getState 作为参数，可以执行异步操作后再派发真正的 action。

javascript

const fetchUser = (id) => async (dispatch) => {
  dispatch({ type: 'FETCH_USER_REQUEST' });
  try {
    const response = await api.getUser(id);
    dispatch({ type: 'FETCH_USER_SUCCESS', payload: response.data });
  } catch (error) {
    dispatch({ type: 'FETCH_USER_FAILURE', error });
  }
};

六、Redux 的设计哲学总结

  • 可预测性：纯函数 reducer + 单一数据源，使得状态变化可追踪、可测试。
  • 可维护性：状态逻辑集中，action 和 reducer 分离，便于多人协作。
  • 生态丰富：中间件机制让扩展变得灵活，如 redux-saga、redux-observable 等处理复杂副作用。
  • 时间旅行：由于状态不可变，可以轻松实现撤销/重做、热重载等调试功能。

七、现代 Redux 的发展

随着 Redux Toolkit（官方工具集）的推出，Redux 的写法更简洁、更推荐。它内置了 configureStore、createSlice、createAsyncThunk 等，减少了样板代码，同时默认集成了 redux-thunk 和 Redux DevTools，并且使用 Immer 库允许在 reducer 中“可变”地更新状态（实际仍不可变）。

但无论写法如何变化，Redux 背后的原理依然是：单向数据流、reducer 纯函数、通过 dispatch action 触发更新。
